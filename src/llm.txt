# Proyecto: Backend para Sistema de Referidos y Fidelización

## Objetivo General

Construir un **backend robusto, escalable y mantenible** para una aplicación de referidos y fidelización, con una arquitectura modular por capas que permita separar la lógica de negocio, facilitar la extensión futura y garantizar la seguridad de los datos de los usuarios.

## Stack Tecnológico

*   **Backend:** Node.js
*   **Framework:** Express.js
*   **ORM:** Sequelize
*   **Base de Datos:** MariaDB (`clarisadb`)
*   **Autenticación:** JWT (JSON Web Tokens)

## Arquitectura por Capas

### Requisitos Funcionales (Foco Actual: Autenticación y Autorización)

1.  **Registro de usuarios (`/api/auth/register`)**

    *   Captura datos personales: nombre, apellido, tipo y número de documento, correo electrónico, contraseña en hash, teléfono.
    *   Validación de unicidad en correo y documento.
    *   Asignación automática de rol base (ejemplo: "referente").
    *   **Si el usuario tiene el rol "referente", se crea automáticamente un perfil de `Referente` asociado**, inicializando sus atributos (código, categoría, estado, etc.).

2.  **Login de usuarios (`/api/auth/login`)**

    *   Autenticación con **JWT**.
    *   Manejo de sesiones sin estado.

3.  **Autorización basada en roles**
    Roles iniciales definidos:

    *   **Admin** → gestión global.
    *   **Referente (cliente interno)** → crea referidos.
    *   **Asesor de ventas** → valida plan de referidos.
    *   **Gerente de ventas** → configura porcentaje de recompensa.
    *   **Contador** → aprueba pagos y retiros.

4.  **Seguridad**

    *   Hash de contraseñas con bcrypt.
    *   Middleware de validación JWT.
    *   Restricciones en endpoints según rol.

## Matriz de Acceso por Roles (Permisos)

Esta matriz define qué acciones puede realizar cada rol en el sistema. Se enfoca en las funcionalidades actuales y las planificadas a corto plazo.

| Recurso / Acción         | Público (No autenticado) | Admin | Referente | Asesor Ventas | Gerente Ventas | Contador |
| ------------------------ | :----------------------: | :---: | :-------: | :-----------: | :------------: | :------: |
| **Autenticación**        |                          |       |           |               |                |          |
| `POST /auth/register`    |            ✅            |   -   |     -     |       -       |       -        |    -     |
| `POST /auth/login`       |            ✅            |   -   |     -     |       -       |       -        |    -     |
|                          |                          |       |           |               |                |          |
| **Usuarios (Users)**     |                          |       |           |               |                |          |
| `GET /users` (Listar)    |            ❌            |   R   |     ❌     |       R       |       R        |    R     |
| `GET /users/:id` (Ver)   |            ❌            |   R   |   Propio  |       R       |       R        |    R     |
| `PUT /users/:id` (Editar)|            ❌            |  R/W  |   Propio  |       ❌       |       ❌        |    ❌     |
| `DELETE /users/:id`      |            ❌            |   W   |     ❌     |       ❌       |       ❌        |    ❌     |
|                          |                          |       |           |               |                |          |
| **Referidos (Refereds)** |                          |       |           |               |                |          |
| `POST /refereds` (Crear) |            ❌            |   W   |     W     |       ❌       |       ❌        |    ❌     |
| `GET /refereds` (Listar) |            ❌            |   R   |   Propios |       R       |       R        |    R     |
| `PUT /refereds/:id`      |            ❌            |  R/W  |     ❌     |      R/W      |       R        |    R     |

**Leyenda:**
*   **R/W**: Lectura y Escritura
*   **R**: Solo Lectura
*   **W**: Solo Escritura
*   **Propio**: Solo puede afectar a los recursos que le pertenecen.
*   **✅**: Acción permitida.
*   **❌**: Acción denegada.
*   **-**: No aplica.

## Diseño de Base de Datos (MariaDB con Sequelize)

### Tabla **Rol**

```sql
CREATE TABLE Rol (
    id_rol INT AUTO_INCREMENT PRIMARY KEY,
    nombre_rol VARCHAR(50) NOT NULL,
    descripcion TEXT,
    creado_en TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

Define los distintos roles dentro del sistema.

### Tabla **Tipo_documento**

```sql
CREATE TABLE Tipo_documento (
    id_tipo_documento INT AUTO_INCREMENT PRIMARY KEY,
    nombre VARCHAR(100) NOT NULL,
    descripcion TEXT
);
```

Permite flexibilidad en la identificación de usuarios (cédula, pasaporte, etc.).

### Tabla **Usuario**

```sql
CREATE TABLE Usuario (
    id_usuario INT AUTO_INCREMENT PRIMARY KEY,
    correo_electronico VARCHAR(150) NOT NULL UNIQUE,
    contrasena_hash VARCHAR(255) NOT NULL,
    nombre VARCHAR(100) NOT NULL,
    apellido VARCHAR(100) NOT NULL,
    numero_documento_identidad VARCHAR(50) NOT NULL UNIQUE,
    telefono VARCHAR(20),
    fecha_registro TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    id_tipo_documento INT NOT NULL,
    CONSTRAINT fk_usuario_tipo_documento
        FOREIGN KEY (id_tipo_documento) REFERENCES Tipo_documento(id_tipo_documento)
        ON UPDATE CASCADE ON DELETE RESTRICT
);
```

Garantiza unicidad en correo y documento. Relaciona cada usuario con un tipo de documento.

### Tabla **Referente**

```sql
CREATE TABLE Referente (
    id_referente INT PRIMARY KEY,
    codigo_referente VARCHAR(20) NOT NULL UNIQUE,
    tipo_referente ENUM('cliente externo', 'cliente interno') DEFAULT 'cliente externo',
    puntos_acumulados INT DEFAULT 0,
    categoria_actual ENUM('basico') DEFAULT 'basico',
    recompensa_monetaria_actual DECIMAL(10, 2),
    fecha_ultima_categoria DATETIME DEFAULT CURRENT_TIMESTAMP,
    estado_referente ENUM('activo', 'en pausa') DEFAULT 'activo',
    CONSTRAINT fk_referente_usuario
        FOREIGN KEY (id_referente) REFERENCES Usuario(id_usuario)
        ON UPDATE CASCADE ON DELETE CASCADE
);
```

Almacena el perfil de un usuario que actúa como referente. Se vincula 1 a 1 con la tabla Usuario.

### Tabla **rol_usuario** (relación N:M Usuario–Rol)

```sql
CREATE TABLE rol_usuario (
    id_usuario INT NOT NULL,
    id_rol INT NOT NULL,
    fecha_asignacion DATETIME DEFAULT CURRENT_TIMESTAMP,
    estado ENUM('activo', 'inactivo') DEFAULT 'activo',
    PRIMARY KEY (id_usuario, id_rol),
    CONSTRAINT fk_rol_usuario_usuario
        FOREIGN KEY (id_usuario) REFERENCES Usuario(id_usuario)
        ON UPDATE CASCADE ON DELETE CASCADE,
    CONSTRAINT fk_rol_usuario_rol
        FOREIGN KEY (id_rol) REFERENCES Rol(id_rol)
        ON UPDATE CASCADE ON DELETE CASCADE
);
```

Permite asignar múltiples roles a un usuario, con control de estado y fecha de asignación.

---

## Relaciones Clave

*   **Usuario – Rol:** N:M mediante `rol_usuario`.
*   **Usuario – Tipo_documento:** 1:N (un tipo de documento puede pertenecer a muchos usuarios).
*   **Usuario – Referente:** 1:1 (un usuario tiene un único perfil de referente).
*   **Usuario (Referente) – Referido:** 1:N (un referente puede tener varios referidos).
*   **Usuario – Reward:** 1:N (un usuario puede tener múltiples recompensas).
